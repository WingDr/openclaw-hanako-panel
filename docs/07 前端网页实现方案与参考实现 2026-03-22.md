# OpenClaw Panel 前端网页实现方案与参考实现

更新日期 2026-03-22

这份文档专门整理 OpenClaw Panel 的前端网页实现方法。  
重点不是继续泛泛讨论，而是把前端应该怎么做、为什么这样做、需要满足什么要求、以及可以参考哪些项目，集中整理成一份可直接指导开发的文档。

## 一句话结论

前端建议做成纯 web SPA。  
最推荐的组合是：

- React
- TypeScript
- Vite
- React Router
- Zustand
- TanStack Query
- TailwindCSS

这套不是最花哨的组合，但它最适合 OpenClaw Panel 这种同时包含 chat workbench 和 manage console 的产品，也最适合后续纯 vibe coding。

## 这份文档要解决什么问题

这份文档主要回答四个问题。

1. OpenClaw Panel 的前端网页到底要实现什么
2. 应该用什么框架和技术栈
3. 前端在产品和工程层面分别有哪些要求
4. 应该如何拆分参考项目，哪些地方学哪个项目

## 一、前端网页到底要实现什么

OpenClaw Panel 的前端不是普通官网，也不是单一的后台表单页。  
它本质上要同时覆盖两类界面。

## 1. Chat Workbench

这一部分是 agent 工作区。  
应该承载这些能力。

- agent 切换
- session 列表
- 新建独立 session
- chat 主界面
- tool stream
- session 元信息
- 右侧上下文信息
- 流式更新

它的核心不是消息收发，而是“持续工作”。  
所以它必须比 Telegram 或 Discord 那种消息窗更适合长期任务协作。

## 2. Manage Console

这一部分是结构化管理区。  
应该承载这些能力。

- overview
- agents
- sessions
- logs
- config
- cron
- skills
- channels
- models

它的核心不是陪伴感，而是信息密度和可操作性。

## 3. 所以这个前端必须同时具备三种能力

- 结构化路由
- 实时状态处理
- 高信息密度 UI

这就是为什么不建议直接用一个简单聊天页思路来做，也不建议纯套 admin template。

## 二、最推荐的前端框架组合

## 1. 第一选择

最推荐的是：

- React
- TypeScript
- Vite
- React Router

## 为什么是它

原因很简单。

- 参考项目最多
- 组件生态最成熟
- AI 最容易生成稳定代码
- 后面想融合 openhanako 风格最顺
- chat 和 manage 双模式都很好做
- 局域网部署时仍然可以保持轻量

## 2. 建议配套组合

在这个基础上，建议搭配：

- Zustand 负责本地 UI 状态和交互状态
- TanStack Query 负责 HTTP 数据获取与缓存
- TailwindCSS 负责样式系统
- Radix UI 或 shadcn 风格组件做基础组件层
- ​`@melloware/react-logviewer` 用作日志页第一版实现

## 3. 为什么不推荐 Next.js

Next.js 当然能做，但第一版不推荐。

原因不是它不好，而是它解决的问题和这个 panel 的核心问题不一致。

- 不需要 SEO
- 首屏 SSR 价值不高
- 核心难点在实时连接，不在服务端渲染
- 如果再叠加 panel proxy，前后端边界会变模糊
- 纯 vibe coding 时，Next 很容易带来不必要的工程复杂度

所以如果只是为了把 OpenClaw Panel 前端做对，Vite 更贴合。

## 4. 为什么不继续跟官方 Lit

OpenClaw 官方 Control UI 用的是 Vite + Lit。  
这没有问题。

但从你的目标看，React 仍然更值得选。

- 参考项目大多是 React
- 组件生态更丰富
- React + Tailwind + Zustand 的 AI 生成成功率更高
- 后面想融 openhanako 风格也更自然

所以这里不是否定 Lit，而是出于实现效率和维护成本做判断。

## 三、前端的产品要求

## 1. chat 和 manage 必须明确分开

这是整个前端最重要的结构要求。

- ​`/chat` 是工作区
- ​`/manage` 是管理区

主题统一，但任务不同。  
不要把所有东西塞在一个大首页里。

## 2. session 必须是一等公民

前端天然要支持：

- 一个 agent 多个 session
- 独立 session 创建
- session pin
- session title/tag
- session 最近访问
- session 与 chat 主界面的联动

## 3. 实时感必须成立

至少下面这些要做到实时或准实时。

- chat delta
- tool 更新
- logs 追加
- session 状态变化
- gateway 连接状态

## 4. 日志页不能只是文本框

日志页至少要支持：

- 自动跟随
- 暂停滚动
- 搜索
- level 过滤
- 大量日志时不卡

## 5. chat 页面不能只是普通消息列表

chat 页面应该承载这些内容。

- tool 时间线
- session 信息
- 上下文提示
- agent 切换
- 独立 session 的归属感

## 四、前端的工程要求

## 1. 路由必须先定清楚

建议最少先定成这样。

```text
/chat
/chat/:agentId/:sessionKey

/manage/overview
/manage/agents
/manage/sessions
/manage/logs
/manage/config
/manage/cron
/manage/skills
/manage/channels
/manage/models
```

## 2. 状态要分层

不要把所有状态都塞在一个 store 里。

建议至少拆成：

- UI 状态
- 会话状态
- 实时事件状态
- manage 数据状态

推荐的做法是：

- Zustand 管 UI 和交互状态
- TanStack Query 管 HTTP 拉取数据
- WebSocket 单独做 event bridge，把增量事件灌进 store

## 3. 主题系统要一开始就搭好

因为已经确定：

- chat 更像 openhanako
- manage 更像结构化控制台

所以主题系统最好一开始就有这些层级。

- design tokens
- color roles
- spacing
- radius
- shadow
- typography
- dark mode

别等页面全部写完了再想统一风格。

## 4. 基础组件和业务组件要分开

### 基础组件

- Button
- Input
- Tabs
- Sidebar
- Card
- Dialog
- Badge

### 业务组件

- SessionList
- ChatComposer
- ToolTimeline
- AgentSwitcher
- LogViewer
- GatewayStatusPill
- SessionMetaPanel

这样后面改风格才不会牵一发动全身。

## 5. 实时协议必须统一入口

前端内部最好只认一个 realtime client。

例如：

- ​`realtime.connect()`
- ​`realtime.subscribeLogs()`
- ​`realtime.watchSession()`
- ​`realtime.sendChat()`

不要让每个页面自己各开一套 socket。

## 五、推荐的前端实现方法

## 第一步

先搭前端骨架，不连真实数据。

优先做这些：

- Layout
- 左侧导航
- 顶栏
- ​`/chat`
- ​`/manage/*`
- 主题 token
- 假数据页面

目标不是功能齐全，而是先把产品骨架和视觉语言定下来。

## 第二步

接 session 和 chat。

优先打通：

- agent switcher
- session list
- session create
- chat main view
- chat stream

这一步最重要。  
因为 chat 是整个产品的灵魂区。

## 第三步

接 manage 基础页。

优先顺序建议：

- overview
- sessions
- logs
- agents

## 第四步

再补控制和配置页。

- config
- cron
- skills
- channels
- models

## 第五步

最后再打磨视觉统一和高级交互。

- pinned sessions
- right panel
- command palette
- log filters
- tool timeline polish
- 页面过渡与细节

## 六、参考实现原则

这里最重要的原则是：  
不要找一个项目完整照抄。  
要按层参考。

## 1. openhanako

适合借：

- 页面氛围
- 工作台感
- chat 区视觉语言
- onboarding 感
- 左侧栏和主内容区的关系

不适合直接搬：

- 内部 managers
- runtime 假设
- memory/desk 底层

## 2. CodePilot

适合借：

- workspace 感
- chat 与项目管理共存方式
- split screen 思路
- session 控制感

## 3. openclaw-dashboard

适合借：

- manage 路由组织方式
- overview 到 detail 的结构
- logs / sessions / memory / config 这种后台页分区方式

## 4. 官方 OpenClaw Control UI

适合借：

- 真实可调用能力边界
- logs 页思路
- sessions / config / skills / cron 的实际数据来源
- 哪些事情本来就是 Gateway 支持的

它不是拿来学视觉的，而是拿来学“哪些东西你不需要重复造”。

## 5. react-logviewer

适合借：

- 大日志显示
- follow 模式
- ANSI 颜色
- 虚拟滚动

它很适合做 `/manage/logs` 的第一版。

## 八、最终推荐

如果现在只认一个前端组合，我建议就定成下面这样。

### 前端框架

- React + TypeScript + Vite

### 路由

- React Router

### 状态

- Zustand

### 数据获取

- TanStack Query

### 样式

- TailwindCSS

### 基础组件

- Radix UI 或 shadcn 风格组件

### 日志页

- 第一版接 `@melloware/react-logviewer`

### 页面结构

- chat 和 manage 分路由

### 开发顺序

- 先静态页面骨架
- 再接真实接口
- 最后做实时流和细节

## 结论

前端网页的最佳实现方式，不是找一个项目整套照抄。  
而是按层参考、按职责拆分。

- 视觉参考 openhanako
- 工作台感参考 CodePilot
- manage 分区参考 openclaw-dashboard
- 数据能力边界参考 OpenClaw 官方 Control UI
- 日志组件参考 react-logviewer

这样做出来的才会是你自己的 OpenClaw Panel。  
而不是别的项目的翻版。
