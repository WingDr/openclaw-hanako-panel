# OpenClaw Panel Proxy 设计稿

更新日期 2026-03-21

这份文档专门整理 panel proxy 的设计。  
目标是给局域网 HTTP 访问场景提供一个可落地的中间层方案，同时保持 OpenClaw Gateway 仍然是唯一的真实控制面。

## 一句话定位

panel proxy 是一个很薄的适配层。

它不拥有 OpenClaw 的核心状态。  
它只负责：

- 给浏览器提供稳定的局域网 HTTP 入口
- 代浏览器连接 OpenClaw Gateway WebSocket
- 把 OpenClaw 能力整理成浏览器更好接的 HTTP 和 WebSocket 接口
- 处理日志转推、鉴权、订阅和少量 panel 自己的展示元数据

## 为什么需要它

如果目标是标准模式，也就是浏览器直连 OpenClaw Gateway WebSocket，并且能使用 HTTPS，那么 panel proxy 不是强制需要。

但如果目标是下面这种使用方式，proxy 就很值得加。

- 局域网内多个设备访问
- 页面用普通 HTTP 打开
- 不想让浏览器直接处理 Gateway 的 pairing、device identity、token 等复杂细节
- 希望 chat、logs、system 状态都通过一个统一通道到前端

## 它该做什么

## 核心职责

### 1. 浏览器接入层

- 提供前端静态页面和普通 HTTP API
- 维护浏览器登录态
- 提供浏览器 WebSocket 实时通道

### 2. Gateway 代理层

- 建立到 OpenClaw Gateway 的 operator WebSocket 连接
- 负责请求转发
- 负责上游事件接收和分发
- 负责连接重试和状态维护

### 3. 实时事件分发

- chat 流式输出
- tool stream
- session 更新
- presence 更新
- logs 更新

### 4. Panel 自己的轻元数据

只存 panel 自己需要的展示信息。

例如：

- session title
- pinned
- tags
- archived
- lastOpenedAt

这些不是 OpenClaw 的权威状态，只是 panel 视图层的补充。

## 它不该做什么

为了避免它长成第二个 OpenClaw，下面这些不应该做。

- 不保存 transcript 作为权威来源
- 不自己实现 session 真相
- 不重写 tool 调度逻辑
- 不把 OpenClaw 的所有 RPC 再包成一套复杂业务系统
- 不第一版就接管安装器、Docker、系统服务、更新器整套能力

一句话说：  
OpenClaw 是大脑。  
panel proxy 只是适配器。

## 推荐总体架构

```text
Browser
  ├─ HTTP
  └─ WebSocket
        ↓
Panel Proxy
  ├─ HTTP API
  ├─ Browser WS Hub
  ├─ Gateway WS Client
  ├─ Log Tail Poller
  └─ Session Meta Store
        ↓
OpenClaw Gateway WebSocket
```

## 推荐技术栈

为了适合纯 vibe coding，建议整个 panel proxy 使用 TypeScript。

### 推荐组合

- Node.js 20+
- Fastify
- ​`@fastify/websocket`
- Zod
- Pino
- 原生 fetch 或 `undici`
- 可选 `nanoid`
- 可选 `better-sqlite3` 或先用 JSON 文件保存 panel 元数据

## 为什么选这套

- 和前端同语言
- 样板少
- Fastify 插件化适合做薄服务
- WebSocket 接入简单
- Zod 很适合统一 HTTP 和 WS 的 schema
- 比 Nest 更轻
- 比 Go 更适合当前阶段的快速试错和 AI 生成代码

## 模块拆分建议

我建议内部至少拆成 6 个模块。

## 1. auth

职责：

- 浏览器到 proxy 的登录认证
- session cookie
- 基础权限检查

首版 建议：

- 单管理员登录
- 同源访问
- 最少量的会话管理

## 2. gateway-client

职责：

- 建立到 OpenClaw Gateway 的 WebSocket 连接
- 处理握手和 request id
- 发送 RPC
- 接收 Gateway events
- 自动重连
- 跟踪 pending requests

## 3. browser-hub

职责：

- 管理浏览器 WebSocket 连接
- 管理订阅关系
- 广播 chat、logs、system 事件
- 将浏览器命令转给 gateway-client 或本地服务

## 4. logs-service

职责：

- 处理 `logs.subscribe`
- 调用 OpenClaw `logs.tail`
- 保存 cursor
- 周期轮询
- 推送 `logs.append`
- 必要时推送 `logs.reset`

## 5. panel-session-meta

职责：

- 保存 panel 自己的 session 展示元数据
- 不和 OpenClaw transcript 混用

字段建议：

- title
- pinned
- tags
- icon
- archived
- lastOpenedAt

## 6. manage-service

职责：

- 提供 overview、agents、sessions、config、skills、cron 等普通 HTTP 数据接口
- 封装部分非实时调用

## 浏览器到 panel proxy 的接口设计

建议分成两类。

### 一类是 HTTP

负责页面初始化和普通读写。

### 一类是 WebSocket

负责所有实时能力。

这样结构最清楚。

# 二、浏览器到 panel proxy 的 WebSocket 设计

建议统一使用一个 envelope 格式。

## 浏览器发命令

```json
{
  "id": "req_123",
  "type": "cmd",
  "method": "chat.send",
  "params": {
    "sessionKey": "agent:main:panel:design-review",
    "text": "帮我继续完善 panel proxy 设计"
  }
}
```

## proxy 回 ack

```json
{
  "id": "req_123",
  "type": "ack",
  "ok": true,
  "result": {
    "accepted": true,
    "runId": "run_abc"
  }
}
```

## proxy 主动推事件

```json
{
  "type": "event",
  "event": "chat.delta",
  "topic": "session:agent:main:panel:design-review",
  "payload": {
    "runId": "run_abc",
    "delta": "好的，我们先从..."
  }
}
```

这样前后端都容易调试和演进。

# 六、日志模块的实现建议

这是 panel proxy 里最关键的模块之一。

## 第一版策略

proxy 内部轮询 `logs.tail`​。  
浏览器只接收 `logs.append`。

## 状态结构建议

```ts
type LogSubscriptionState = {
  source: "gateway";
  cursor: number | null;
  subscribers: Set<string>;
  polling: boolean;
  lastPollAt: number | null;
};
```

## 基本流程

### 1. 浏览器发 `logs.subscribe`

### 2. proxy 发现这是第一个订阅者

启动轮询器

### 3. 每 1000 到 2000ms 调一次 `logs.tail`

参数带上：

- ​`cursor`
- ​`limit`
- ​`maxBytes`

### 4. 如果有新日志

广播：

```json
{
  "type": "event",
  "event": "logs.append",
  "topic": "logs:gateway",
  "payload": {
    "lines": ["...", "..."],
    "cursor": 12345
  }
}
```

### 5. 如果 cursor 丢失或需要重置

推送：

```json
{
  "type": "event",
  "event": "logs.reset",
  "topic": "logs:gateway",
  "payload": {
    "reason": "cursor-invalid"
  }
}
```

## 前端日志页收到事件后的处理

- 追加到本地 buffer
- 应用 level filter
- 应用文本搜索
- 如果 auto-follow 开启就滚到底
- buffer 超过上限就裁掉旧行

# 七、session key 设计

建议 panel proxy 生成的独立 session key 统一用这几种形式。

- ​`agent:<agentId>:panel:<slug>`
- ​`agent:<agentId>:task:<slug>`
- ​`agent:<agentId>:workspace:<slug>`

先不要发明更复杂的 schema。

例如：

- ​`agent:main:panel:design-review`
- ​`agent:main:task:panel-mvp`
- ​`agent:research:workspace:paper-notes`

# 十、最终推荐

如果现在就准备开工，我推荐这样定。

## 第一版架构

- 前端是 web-only
- 局域网访问走 HTTP
- 中间加一个 TypeScript 的薄 panel proxy
- 浏览器对 proxy 用：

  - HTTP 做普通请求
  - 一条 WebSocket 做所有实时能力
- proxy 对 OpenClaw 用：

  - WebSocket 直连 Gateway
  - ​`logs.tail` 轮询做日志转推

## 第一版优先做的接口

### HTTP

- ​`/api/bootstrap`
- ​`/api/agents`
- ​`/api/sessions`
- ​`/api/chat/:sessionKey/history`
- ​`/api/overview`
- ​`/api/config`
- ​`/api/cron`

### WS 命令

- ​`session.create`
- ​`chat.send`
- ​`chat.abort`
- ​`logs.subscribe`
- ​`session.watch`

### WS 事件

- ​`chat.delta`
- ​`tool.update`
- ​`logs.append`
- ​`session.updated`
- ​`system.connection`

## 结论

panel proxy 的最佳定位不是新的业务后端。  
而是一个局域网 HTTP 场景下的薄适配层。

它的价值在于：

- 屏蔽浏览器直连 Gateway 的复杂度
- 提供统一的浏览器实时通道
- 让 logs、chat、session 和 manage 页面更容易实现
- 让整个 panel 更适合纯 vibe coding 的开发方式
