# OpenClaw Panel 首版前端页面树与组件结构

更新日期 2026-03-22

这份文档的目标不是继续讨论“做什么”，而是把首版前端该怎么落成一个可以直接开工的页面骨架说明清楚。

它只服务当前已经收束过的最小范围：

- 分 agent chat
- 实时 log 监控
- gateway / agent / channel 状态监控

不覆盖完整 manage 面板。
不覆盖复杂 design token 系统。
不覆盖后续 phase 2 的 config / cron / skills 等扩展页。

## 一句话结论

首版前端先固定成 3 个页面：

- `/chat`
- `/logs`
- `/status`

并在代码结构上拆成 4 层：

- routes
- components
- store
- lib

目标是先把 **页面骨架、交互骨架、实时数据入口** 固定下来，让后续接代理层和 Gateway 数据时不需要返工信息架构。

## 一、设计目标

这份设计稿要解决 4 件事：

1. 前端第一版到底有哪些页面
2. 每个页面的职责是什么
3. 组件应该怎么拆，避免一上来就写成大组件
4. 哪些状态放 store，哪些状态只留在组件局部

## 二、实现边界

## 只做什么

### `/chat`

- agent 列表
- 当前 agent 的 session 列表
- 新建独立 session
- 打开 session
- 消息流
- 简化版 tool 状态区域
- 输入框与发送/中止

### `/logs`

- 日志快照展示
- follow 开关
- pause / resume
- level filter
- 文本搜索
- buffer 裁剪

### `/status`

- gateway 连接状态
- agent 状态卡片
- channel 状态卡片
- recent sessions 列表

## 先不做什么

- `/manage/*`
- session pin / tag / archive
- 右侧复杂上下文面板
- 命令面板
- 分屏
- 多用户
- 动态主题切换

## 四、页面树

## 1. 顶层路由

```text
/
  -> redirect /chat

/chat
/logs
/status
```

## 2. 顶层壳结构

三个页面都共用同一个基础壳：

```text
AppShell
  ├─ SidebarNav
  ├─ Topbar
  └─ MainContent
```

### SidebarNav 首版只放

- Chat
- Logs
- Status

### Topbar 首版只放

- 当前页面标题
- Gateway 连接状态
- 当前选中 agent（若页面需要）

不要在首版顶栏里塞：

- 全局搜索
- 用户菜单
- 通知中心
- command palette 入口

## 五、每个页面的结构与职责

## 1. `/chat`

### 页面职责

- 让用户围绕某个 agent 与某个 session 持续工作
- 展示聊天流和最小化 tool 状态
- 提供 session 创建和切换入口

### 页面结构

```text
/chat
  AppShell
    PageHeader("Chat")
    ChatPage
      ├─ AgentList            // 左栏
      ├─ SessionList          // 中栏
      └─ ChatWorkspace        // 主区
           ├─ ChatView
           │   ├─ ChatMessageList
           │   └─ ToolStatusStrip
           └─ ChatComposer
```

### 布局建议

```text
┌────────────┬──────────────┬──────────────────────────────┐
│ Agents     │ Sessions     │ Chat                         │
│            │              │                              │
│ main       │ panel/a      │ message list                 │
│ research   │ panel/b      │ tool status                  │
│ design     │ + new        │ composer                     │
└────────────┴──────────────┴──────────────────────────────┘
```

### 组件职责

#### `AgentList`

- 展示 agent 列表
- 高亮当前选中 agent
- 切换 agent 时触发 session 列表加载

#### `SessionList`

- 展示当前 agent 下的 session 列表
- 打开 session
- 触发“新建 session”对话框

#### `SessionCreateDialog`

- 收集：`agentId`、`slug`、可选 `title`
- 预览生成后的 `sessionKey`
- 创建成功后自动切到新 session

#### `ChatMessageList`

- 显示历史消息和当前流式消息
- 支持滚动到底部
- streaming 时自动 follow

#### `ToolStatusStrip`

- 只展示当前 run 的简化 tool 状态
- 不做完整时间线
- 建议字段：`toolName`、`status`、`summary`

#### `ChatComposer`

- 文本输入
- 发送按钮
- streaming 中止按钮
- 禁用态处理

### 页面最小状态依赖

- `selectedAgentId`
- `selectedSessionKey`
- `agents`
- `sessionsByAgent`
- `chatMessagesBySession`
- `streamingBySession`
- `currentToolStateBySession`

## 2. `/logs`

### 页面职责

- 展示 Gateway 日志
- 管理日志订阅与本地过滤
- 保持日志展示流畅，不追求第一版全功能

### 页面结构

```text
/logs
  AppShell
    PageHeader("Logs")
    LogsPage
      ├─ LogToolbar
      └─ LogPanel
```

### 布局建议

```text
┌────────────────────────────────────────────────────────────┐
│ level filter | search | follow | pause/resume | clear     │
├────────────────────────────────────────────────────────────┤
│ log rows...                                                │
│ log rows...                                                │
│ log rows...                                                │
└────────────────────────────────────────────────────────────┘
```

### 组件职责

#### `LogToolbar`

- level 多选
- 文本搜索
- follow 开关
- pause / resume
- clear local buffer

#### `LogPanel`

- 渲染日志列表
- 应用过滤结果
- follow 开启时滚到底部
- buffer 超上限时裁剪旧日志

#### `LogLineRow`

- 根据 level 显示颜色
- 保持单行或软换行策略一致

### 页面最小状态依赖

- `logs.lines`
- `logs.follow`
- `logs.paused`
- `logs.levels`
- `logs.search`
- `logs.cursor`
- `logs.connected`

## 3. `/status`

### 页面职责

- 展示系统健康概览
- 提供一个不需要进入 chat / logs 的总览入口

### 页面结构

```text
/status
  AppShell
    PageHeader("Status")
    StatusPage
      ├─ GatewayStatusCard
      ├─ AgentStatusGrid
      ├─ ChannelStatusGrid
      └─ RecentSessionsCard
```

### 布局建议

```text
┌──────────────────────────┐
│ GatewayStatusCard        │
├──────────────────────────┤
│ AgentStatusGrid          │
├──────────────────────────┤
│ ChannelStatusGrid        │
├──────────────────────────┤
│ RecentSessionsCard       │
└──────────────────────────┘
```

### 组件职责

#### `GatewayStatusCard`

- 显示连接状态
- 最近心跳 / 更新时间
- 连接异常时高亮

#### `AgentStatusGrid`

- 渲染 agent 列表与状态
- 字段建议：`agentId`、`status`、`lastSeenAt`

#### `ChannelStatusGrid`

- 渲染 channel 列表与状态
- 字段建议：`channelKey`、`status`、`summary`

#### `RecentSessionsCard`

- 展示最近活跃 sessions
- 字段建议：`sessionKey`、`agentId`、`updatedAt`、`preview`

## 六、Store 设计

首版先用一份 Zustand store 即可，但内部结构要按职责分块。

```ts
type AppState = {
  ui: {
    currentRoute: "/chat" | "/logs" | "/status";
    selectedAgentId: string | null;
    selectedSessionKey: string | null;
  };

  chat: {
    agents: AgentSummary[];
    sessionsByAgent: Record<string, SessionSummary[]>;
    messagesBySession: Record<string, ChatMessage[]>;
    streamingBySession: Record<string, StreamingState | null>;
    toolStateBySession: Record<string, ToolStateSummary | null>;
  };

  logs: {
    lines: LogLine[];
    cursor: number | null;
    follow: boolean;
    paused: boolean;
    levels: string[];
    search: string;
    connected: boolean;
  };

  status: {
    gatewayConnected: boolean;
    lastUpdatedAt: string | null;
    snapshot: StatusSnapshot | null;
  };
};
```

## 七、状态分层规则

## 放到 store 的状态

- 当前 agent / session 选择
- session 列表
- chat 消息
- 流式状态
- logs buffer
- status snapshot
- 连接状态

## 放到组件局部的状态

- 对话框开关
- 输入框草稿
- 临时 hover / focus 状态
- 局部滚动位置

原则是：

**跨页面、跨组件共享的才进 store；只影响单个组件展示的留在局部。**

## 八、API 与实时入口分工

## `api.ts` 负责

- `GET /api/bootstrap`
- `GET /api/agents`
- `GET /api/agents/:agentId/sessions`
- `GET /api/status`
- `GET /api/logs/snapshot`

## `realtime.ts` 负责

- 建立浏览器实时连接
- 发送 `chat.send`
- 发送 `chat.abort`
- 发送 `session.create`
- 发送 `session.open`
- 发送 `logs.subscribe`
- 发送 `logs.unsubscribe`

## `eventBridge.ts` 负责

- 把 socket 事件转换为 store action
- 处理 `chat.delta`、`chat.done`、`logs.append`、`status.snapshot` 等事件

## 九、页面初始化顺序

## App 启动

1. 加载 `bootstrap`
2. 初始化 realtime client
3. 拉 `agents`
4. 如果没有用户选择，自动选默认 agent

## 进入 `/chat`

1. 拉当前 agent 的 sessions
2. 默认选第一个 session；如果没有则展示 empty state
3. 打开 session 时拉历史

## 进入 `/logs`

1. 拉 `logs snapshot`
2. 发送 `logs.subscribe`
3. 离开页面时可保留连接，也可暂停订阅，二选一固定实现

## 进入 `/status`

1. 拉 `status`
2. 启动定时刷新或等待 `status.snapshot`

## 十、执行顺序建议

### 第一步：静态页面骨架

- AppShell
- 三个 route
- 假数据卡片

### 第二步：接 HTTP 数据

- agents
- sessions
- status
- logs snapshot

### 第三步：接实时事件

- chat send / delta / done
- logs subscribe / append
- system.connection / status.snapshot

### 第四步：补交互细节

- loading
- error
- reconnect
- empty state
- buffer 裁剪

## 十一、完成标准

满足下面这些条件，就算这份页面树设计已经被正确实现：

1. 三个页面都能独立访问
2. `/chat` 能完成 agent 切换、session 切换、发消息、收流
3. `/logs` 能展示快照并持续追加日志
4. `/status` 能稳定展示 gateway / agent / channel / recent sessions
5. 组件没有退化成一个超大页面文件
6. 实时连接入口只有一套

## 结论

首版前端的关键不是页面多，而是：

- 页面职责清楚
- 组件拆分稳定
- store 边界明确
- 实时入口统一

这样后面从 `/chat + /logs + /status` 扩展到完整 `/manage/*` 时，结构才不会推倒重来。
