# 对话记录整理

整理日期 2026-03-21

这份记录不是原始逐字转录。  
它是把这轮关于 OpenClaw Panel 的讨论整理成时间线和决策轨迹，方便以后回看为什么做出这些判断。

## 时间线

## 1. 最初起点

用户先提出一个很核心的问题。

- 参考 `openhanako`
- 判断仿照它的前端风格做一个 OpenClaw control panel 是否可行

这一阶段的重点是确认大方向。

### 形成的初步判断

- 可以做
- openhanako 适合参考前端气质和工作台感
- 不能直接照搬它的内部系统模型

## 2. 引入 ClawPanel

随后讨论转向 ClawPanel。

用户希望分析。

- ClawPanel 是怎么调用 OpenClaw 的
- 能否融合进 openhanako 风格前端
- 能否加入独立 session

### 形成的判断

- ClawPanel 更像一个 OpenClaw 管理外壳
- 它大量封装了配置和进程管理，不只是聊天界面
- OpenClaw 的真正控制面是 Gateway WebSocket
- 独立 session 本质上是 session key 设计问题，而不是频道问题

## 3. 第一批正式文档

在这个阶段，开始把前面分析整理为正式文档。  
同时也确认了最早写下来的旧稿只能当草稿，不应当成最终结论。

### 形成的文档层次

- 初步开发方案草稿
- 已验证的调研结论稿

### 形成的判断

- 最接近 OpenClaw 原生架构的方式，是前端直连 Gateway WS
- 独立 session 可以通过 agent 级 session key 实现

## 4. 扩展参考 OpenClaw 面板类项目

之后又加入了几类 OpenClaw 相关管理界面参考。

- ​`openclaw-agent-monitor`
- ​`openclaw-manager`
- ​`openclaw-dashboard`

用户提出了一个重要的产品思路。

- 页面尽量轻量但结构化
- chat 部分和 manage 部分分开
- 通过路由切换
- 主题保持一致

### 形成的判断

这是整轮讨论里最关键的产品结论之一。

- ​`chat` 是工作区
- ​`manage` 是管理区
- 两者主题统一
- 但交互目标必须拆开

## 5. 收束成 web-only 方案

接着用户明确要求。

- 不要 Electron
- 只要 web 版
- 把这套思路落成文档

### 形成的稳定结论

- panel 做成 web-only
- 统一主题
- ​`chat`​ 与 `manage` 分路由
- 独立 session 做成核心能力
- 如果后面需要桌面壳，再后加

## 6. 第二轮更广泛调研

随后用户提出第二轮需求。

1. 继续广泛搜集类 openhanako 的项目
2. 结合 OpenClaw 控制架构决定 panel 的实现架构
3. 研究实时日志展示怎么做

这一轮扩展了更多参考项目。

- openhanako
- Zuckerman
- ai-browser
- CodePilot
- Goose
- VoltAgent / VoltOps Console
- 以及前面几种 OpenClaw 专用 dashboard

### 形成的稳定结论

- 最适合的前端技术路线是 `React + TypeScript + Vite`
- 路由建议使用 `React Router`
- 状态层建议使用 `Zustand`
- 数据层建议用 `TanStack Query`
- 样式建议用 `TailwindCSS`
- chat、session、tool stream 应优先直连 OpenClaw Gateway WS
- 实时日志第一版直接复用 `logs.tail`

## 7. 局域网访问问题

之后用户提出一个非常实际的问题。

- 如果希望局域网访问，是否还能让前端直连 WebSocket

进一步收束后，用户明确自己的目标是。

- 准备走局域网 HTTP 访问
- 希望确认是否应该加一个薄网关
- 以及网页该怎么从薄网关拿日志

### 形成的稳定结论

这一轮让架构进一步分化成了两档。

#### 标准模式

- 浏览器直连 OpenClaw Gateway WebSocket
- 适合本机访问，或已有 HTTPS 的环境

#### 局域网 HTTP 模式

- 浏览器不直连 Gateway
- 中间增加一个很薄的 panel proxy
- proxy 对内连接 OpenClaw Gateway WS
- 浏览器对 proxy 用 HTTP + 一条统一 WebSocket
- logs 由 proxy 通过 `logs.tail` 增量拉取，再推送给前端

## 当前总决策

截至这次整理，已经形成的稳定判断如下。

## 产品层

- 做 web-only panel
- 采用 chat 与 manage 分路由
- 统一主题
- 不做 Electron

## 数据链路

### 标准模式

- Browser 直连 OpenClaw Gateway WS

### 局域网 HTTP 模式

- Browser 连 panel proxy
- panel proxy 连 OpenClaw Gateway WS

## session

- 支持 agent 级独立 session
- 推荐 key 形式包括

  - ​`agent:<agentId>:panel:<slug>`
  - ​`agent:<agentId>:task:<slug>`
  - ​`agent:<agentId>:workspace:<slug>`

## 日志

- 第一版不发明新的日志协议
- 直接复用 OpenClaw `logs.tail`
- 如果是 proxy 模式，由 proxy 做 `logs.tail` 轮询并转推

## 最推荐的前端技术组合

- React
- TypeScript
- Vite
- React Router
- Zustand
- TanStack Query
- TailwindCSS
