# Proxy 鉴权机制 2026-03-25

更新日期 2026-03-25

这份文档专门固定当前仓库已经落地的 `panel-web <-> panel-proxy` 鉴权机制。

目标不是引入完整用户系统，
而是在保持 `panel-web` 纯静态前端的前提下，解决两件事：

- `panel-proxy` 不能裸奔，不能让任何人直接调用业务 API / WebSocket
- `curl`、脚本、集成测试仍然要能单独调试 `panel-proxy`

## 一句话结论

当前采用**双轨鉴权**：

- 给人类用户：`panel 密码登录 -> proxy 签发短期 HttpOnly session cookie`
- 给脚本和调试：`Bearer API token`

只要不是认证接口，所有 `/api/*` 和 `/ws` 都必须带着这两类凭证之一。

## 一、为什么不把长期 token 直接塞给前端

`panel-web` 当前是纯静态前端。

这意味着：

- 如果把“长期 machine token”打进前端构建产物，它并不真正保密
- 浏览器里任何能打开页面的人，本质上都能看到这个 token

所以浏览器侧最合理的方案不是“隐藏一个永久 token”，
而是：

1. 用户输入 panel 密码
2. `panel-proxy` 校验密码
3. `panel-proxy` 签发一个短期 session cookie
4. 后续 HTTP / WS 都依赖这个短期会话

这样仍然是 token/session 模型，但 secret 不会被硬编码进前端包。

## 二、两套凭证的职责

### 1. `PANEL_LOGIN_PASSWORD_HASH`

用途：

- 只给浏览器里的 panel 登录使用
- 用户输入的是明文密码
- `proxy` 侧用 `scrypt` 校验明文密码与哈希是否匹配

校验成功后：

- `proxy` 生成短期 session
- 写入 `HttpOnly`、`SameSite=Strict` cookie
- 后续 `panel-web` 的 HTTP 和 WebSocket 都自动复用这个会话

### 2. `PANEL_PROXY_API_TOKEN`

用途：

- 给 `curl`
- 给脚本
- 给集成测试
- 给手工调试 WebSocket 握手

使用方式：

- `Authorization: Bearer <PANEL_PROXY_API_TOKEN>`

它不会注入到 `panel-web`，也不会进入浏览器构建产物。

## 三、接口级规则

### 公开接口

只有下面三个接口是公开的：

- `GET /api/auth/me`
- `POST /api/auth/login`
- `POST /api/auth/logout`

### 受保护接口

其余所有业务接口都必须满足下面任一条件：

- 请求里带有效 session cookie
- 请求里带有效 `Authorization: Bearer <PANEL_PROXY_API_TOKEN>`

包括：

- `GET /api/bootstrap`
- `GET /api/agents`
- `GET /api/agents/:agentId/sessions`
- `GET /api/chat/:sessionKey/history`
- `GET /api/status`
- `GET /api/logs/snapshot`
- `GET /ws`

### 失败约定

HTTP：

```json
{
  "ok": false,
  "error": {
    "code": "unauthorized",
    "message": "Authentication required"
  }
}
```

WebSocket：

- 握手阶段直接拒绝
- 或用 `1008` 关闭连接

## 四、浏览器登录流

### 1. 初始化

`panel-web` 启动后先调用：

- `GET /api/auth/me`

它用于判断：

- 当前是否启用了鉴权
- 当前浏览器是否已经登录
- 当前是否配置了 panel 登录密码

### 2. 登录

用户在登录页输入 panel 密码后，浏览器调用：

- `POST /api/auth/login`

请求体：

```json
{
  "password": "plain-text-password"
}
```

`proxy` 侧做的事情：

1. 读取 `PANEL_LOGIN_PASSWORD_HASH`
2. 解析 `scrypt$N$r$p$salt$hash`
3. 用输入密码重新计算 `scrypt`
4. 与存储 hash 做安全比较
5. 成功后生成 session 并写 cookie

### 3. 已登录状态

登录成功后：

- 浏览器的 `fetch` 通过 `credentials: include` 自动带 cookie
- 浏览器连接 `/ws` 时也自动使用同站 cookie

### 4. 退出

退出调用：

- `POST /api/auth/logout`

`proxy` 会：

- 删除内存 session
- 返回清空后的 cookie

## 五、脚本 / 调试流

HTTP 调试示例：

```bash
curl -H "Authorization: Bearer $PANEL_PROXY_API_TOKEN" \
  http://127.0.0.1:22846/api/bootstrap
```

先登录再带 cookie 的示例：

```bash
curl -c /tmp/panel.cookie \
  -H "Content-Type: application/json" \
  -d '{"password":"你的 panel 密码"}' \
  http://127.0.0.1:22846/api/auth/login

curl -b /tmp/panel.cookie \
  http://127.0.0.1:22846/api/bootstrap
```

WebSocket 调试：

- 使用支持自定义握手 Header 的工具
- 握手时带 `Authorization: Bearer <PANEL_PROXY_API_TOKEN>`

## 六、配置项

### `PANEL_LOGIN_PASSWORD_HASH`

- 浏览器 panel 登录密码哈希
- 当前格式：`scrypt$N$r$p$salt$hash`

### `PANEL_PROXY_API_TOKEN`

- `panel-proxy` 的 Bearer API token

### `PANEL_SESSION_TTL_MS`

- panel 登录 session 的有效期

## 七、运维与开发命令

在仓库根目录执行：

生成并直接写入根 `.env`：

```bash
npm run password:hash
```

或：

```bash
npm run password:hash -- "你的密码"
```

只打印不写入：

```bash
npm run password:hash -- --print-only "你的密码"
```

校验当前 `.env` 的密码哈希是否匹配：

```bash
npm run password:verify
```

或：

```bash
npm run password:verify -- "你的密码"
```

## 八、当前边界

这套方案当前明确不做：

- 多用户
- 角色权限
- API scope
- 持久化 session 存储
- 多实例共享 session

当前只解决“谁能访问这个 proxy”。

## 九、与未来演进的关系

如果后续要引入更复杂的安全模型，推荐的演进方向是：

1. 保持浏览器登录 session 模式不变
2. 再补独立的脚本 token 管理
3. 如果真的需要多用户，再把 `panel-web` 升级成带服务端的完整应用

在那之前，这套双轨模型已经足够覆盖首版局域网 / 本机使用场景。
