# OpenClaw Panel Gateway WS 客户端数据流草案

更新日期 2026-03-22

这份文档定义 首版 阶段浏览器或 proxy 内部的 Gateway WS client 应该如何组织数据流。

重点不是 OpenClaw 协议的完整复述，
而是把 panel 侧真正需要实现的：

- 连接生命周期
- request / response 映射
- event 分发
- chat 流
- logs 调用
- 状态同步

整理成一个可以直接实现的客户端草案。

## 一句话结论

不管 Gateway client 运行在浏览器还是 panel proxy 内部，首版 都应该遵循同一个最小模型：

- 一条长连接
- 一个 requestId 映射表
- 一个 event 分发中心
- chat / logs / status 三条独立业务流

不要把 chat 事件、日志事件、状态事件混成一套巨大的页面逻辑。

## 一、适用范围

这份草案同时适用于两种模式。

### 标准模式

- Browser 直接连接 OpenClaw Gateway WS

### 局域网 HTTP 模式

- panel proxy 内部连接 OpenClaw Gateway WS
- Browser 只连接 panel proxy

这两种模式唯一变化的是：

- **Gateway client 放在哪一层**

它的内部结构不应该变化太大。

## 二、客户端职责

Gateway client 只负责 6 件事：

1. 建立与维护 WebSocket 连接
2. 处理握手与连接状态
3. 发送 RPC 请求并匹配响应
4. 接收事件并分发给上层
5. 管理重连与 pending request 失败
6. 提供统一的业务方法封装

它不负责：

- 页面渲染
- store 更新细节
- 复杂业务规则
- panel 元数据持久化

## 三、最小模块结构

```text
gateway-client/
  connection.ts      // socket connect / reconnect / heartbeat state
  rpc.ts             // request id / pending promise map
  events.ts          // event emitter / topic dispatch
  methods.ts         // chat.send, sessions.list, logs.tail 等封装
  types.ts           // 请求、响应、事件类型
```

如果不想拆太细，首版 也至少要保证代码逻辑上分成这 4 类职责。

## 四、连接状态机

建议显式维护连接状态，而不是靠页面猜。

```ts
type GatewayConnectionState =
  | "idle"
  | "connecting"
  | "authenticating"
  | "connected"
  | "reconnecting"
  | "disconnected"
  | "error";
```

## 推荐状态流

```text
idle
  -> connecting
  -> authenticating
  -> connected
  -> reconnecting (on close / transient failure)
  -> connected

connected
  -> disconnected (manual close)

connecting/authenticating/reconnecting
  -> error (fatal)
```

## 五、统一消息模型

首版 客户端内部建议统一成三种消息：

- request
- response
- event

可以抽象成：

```ts
type GatewayRequest = {
  id: string;
  method: string;
  params?: unknown;
};

type GatewayResponse = {
  id: string;
  ok: boolean;
  result?: unknown;
  error?: {
    code: string;
    message: string;
    details?: unknown;
  };
};

type GatewayEvent = {
  event: string;
  topic?: string;
  payload?: unknown;
};
```

真实协议字段命名可以贴近 OpenClaw，
但 panel 内部一定要把“请求响应”和“推送事件”分开处理。

## 六、pending request 机制

客户端必须维护一个 `requestId -> pending promise` 的映射。

```ts
type PendingRequest = {
  method: string;
  createdAt: number;
  resolve: (value: unknown) => void;
  reject: (error: Error) => void;
  timeoutId: ReturnType<typeof setTimeout>;
};
```

## 发送请求流程

1. 生成 `requestId`
2. 创建 pending entry
3. 发送 request
4. 等待 response
5. 收到后 resolve / reject
6. 清理 timeout 与 map entry

## 连接断开时

如果 socket 断开：

- 所有 pending request 都要 reject
- 错误建议统一成 `GatewayDisconnectedError`

不要让上层 promise 永远挂住。

## 七、业务方法最小集合

首版阶段最少实现这些方法。

## 1. chat

- `chatSend(sessionKey, text)`
- `chatAbort(runId)`
- `chatHistory(sessionKey)`

## 2. sessions

- `sessionsList(agentId?)`
- `sessionCreate({ agentId, slug, kind })`
- `sessionOpen(sessionKey)`

注意：

`sessionCreate` 在 panel 侧可以先表现为一个 helper：
生成 `agent:<agentId>:hanako-panel:<uuid>`，先在 panel 本地建立 `pending` session，再通过第一次 `chat.send` 让上下文真正落地。

## 3. logs

- `logsTail({ cursor, limit, maxBytes })`

## 4. status

- `systemPresence()`
- `statusGet()`
- `modelsList()`（可选）

## 八、事件分发模型

不要让页面直接操作 socket 原始事件。
中间必须有一层 event dispatcher。

```ts
type GatewayEventHandler = (event: GatewayEvent) => void;

type GatewayEventBus = {
  on(eventName: string, handler: GatewayEventHandler): () => void;
  emit(event: GatewayEvent): void;
};
```

## 建议分的业务 topic

- `chat`
- `tool`
- `session`
- `system`
- `logs`

## 映射规则建议

### chat 相关事件

- `chat.started`
- `chat.delta`
- `chat.done`
- `chat.error`
- `chat.aborted`

### tool 相关事件

- `tool.started`
- `tool.updated`
- `tool.done`

### session 相关事件

- `session.updated`

### system 相关事件

- `system.connection`
- `system.presence`
- `status.snapshot`

### logs 相关事件

对于标准模式，如果浏览器直接连 Gateway，日志可以仍然表现为本地轮询结果进入日志 store；
如果走 proxy，则浏览器侧收到的是 proxy 转发的 `logs.append` / `logs.reset`。

## 九、三条核心业务流

## 1. Chat 数据流

```text
UI send
  -> gatewayClient.chatSend()
  -> Gateway request accepted
  -> event: chat.started
  -> event: chat.delta (0..n)
  -> event: tool.started / tool.updated / tool.done (0..n)
  -> event: chat.done
  -> store finalize message
```

### Chat store 更新规则

#### `chat.started`

- 创建一个 streaming run 状态
- 为当前 session 建立“正在生成”的占位消息

#### `chat.delta`

- 追加到当前 streaming message

#### `tool.*`

- 更新 `toolStateBySession[sessionKey]`

#### `chat.done`

- 把 streaming message 固化为普通消息
- 清空当前 run 状态

#### `chat.error`

- streaming 状态结束
- 当前 run 标记错误

## 2. Logs 数据流

### 标准模式建议

浏览器内本地轮询：

```text
page open
  -> logsTail(cursor=null)
  -> snapshot lines
  -> start interval
  -> logsTail(cursor=lastCursor)
  -> append lines
```

### Proxy 模式建议

proxy 轮询 Gateway，浏览器只收事件：

```text
browser subscribe
  -> proxy starts/joins poller
  -> proxy calls logsTail(cursor)
  -> proxy emits logs.append
  -> browser appends buffer
```

### Logs store 更新规则

- `append`: 追加到 buffer
- `reset`: 清空或重置 cursor 后重拉快照
- `paused=true`: 仍可接收数据，但 UI 不自动滚动
- 超过上限时丢弃最旧的 N 行

## 3. Status 数据流

推荐 status 独立于 chat 和 logs。

### 方案 A：纯拉取

- 页面首次加载拉一次
- 每 5 到 10 秒刷新

### 方案 B：拉取 + 事件补丁

- 首次拉一次
- 连接状态靠 `system.connection`
- 其余概览靠 `status.snapshot`

首版可以先做 A，再扩 B。

## 十、重连策略

重连一定要有固定规则，不要散落在页面里。

### 建议参数

- 首次延迟：1000ms
- 最大延迟：10000ms
- 回退：指数退避
- 成功连接后重置 backoff

### 重连时要做的事

1. 更新 connection state 为 `reconnecting`
2. 清空 pending request
3. 重建 socket
4. 触发上层重新同步关键状态

### 重连后上层建议补的同步

- 当前 agent sessions
- 当前打开 session 的 history / latest state
- logs cursor 或 snapshot
- status snapshot

## 十一、错误模型

建议统一成这几类错误：

- `GatewayDisconnectedError`
- `GatewayTimeoutError`
- `GatewayProtocolError`
- `GatewayAuthError`
- `GatewayRpcError`

这样页面层才容易做稳定提示。

## 十二、浏览器侧最小接口示意

```ts
type GatewayClient = {
  connect(): Promise<void>;
  disconnect(): Promise<void>;
  getConnectionState(): GatewayConnectionState;
  on(eventName: string, handler: GatewayEventHandler): () => void;

  chatSend(params: { sessionKey: string; text: string }): Promise<{ runId?: string }>;
  chatAbort(params: { runId: string }): Promise<void>;
  chatHistory(params: { sessionKey: string }): Promise<ChatMessage[]>;

  sessionsList(params?: { agentId?: string }): Promise<SessionSummary[]>;
  logsTail(params: { cursor?: number | null; limit?: number; maxBytes?: number }): Promise<LogsTailResult>;
  systemPresence(): Promise<SystemPresenceResult>;
  statusGet(): Promise<StatusSnapshot>;
};
```

## 十三、完成标准

满足下面这些条件，就可以认为 Gateway client 的数据流设计是可执行的：

1. 所有请求都有 requestId 与 pending map
2. 所有响应都能 resolve / reject 到对应调用方
3. 所有事件都先进入 event bus，再进入 store
4. chat、logs、status 三条流彼此独立
5. 断线不会留下悬挂 promise
6. 重连后能恢复关键页面状态

## 结论

Gateway client 的关键不是“会连 WebSocket”这么简单，
而是把连接、RPC、事件、业务流分层清楚。

只有这样，前端或 proxy 才不会在接入 chat、logs、status 三类数据后迅速变成一团乱逻辑。
