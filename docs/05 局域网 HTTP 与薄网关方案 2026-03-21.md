# 局域网 HTTP 与薄网关方案

更新日期 2026-03-21

这份文档是对局域网访问场景的补充说明。  
前面的文档默认更偏向标准模式，也就是浏览器直接连接 OpenClaw Gateway WebSocket。  
但如果目标是局域网内通过普通 HTTP 页面访问，那么最稳的方案需要调整。

## 一句话结论

如果准备走局域网 HTTP 访问，推荐增加一个很薄的 panel proxy。

- 浏览器不直接连接 OpenClaw Gateway
- panel proxy 负责和 OpenClaw Gateway 建立 WebSocket 连接
- 浏览器通过 HTTP 和一个统一 WebSocket 连接 panel proxy
- 实时日志也由 panel proxy 转推给浏览器

## 为什么不推荐局域网 HTTP 下让浏览器直接连 Gateway

原因不是 WebSocket 不能连，而是整体体验和安全边界会变差。

主要问题在这些地方。

- 局域网远程访问通常不是 localhost
- 纯 HTTP 场景下浏览器不在 secure context
- OpenClaw 浏览器控制面还涉及 device identity、pairing、token 这类控制链路
- 在纯 HTTP + 局域网多设备场景下，浏览器直连 Gateway 的配对和鉴权摩擦会明显增大

所以如果目标是让同网段设备更自然地访问 panel，而不是折腾 Gateway 浏览器侧配对细节，那么中间加一层薄网关是更现实的做法。

## 推荐三层结构

```text
Browser
  ├─ HTTP 请求页面和普通数据
  └─ WebSocket 接收实时事件
        ↓
Panel Proxy
  ├─ 管理浏览器侧会话
  ├─ 转发请求到 OpenClaw Gateway
  ├─ 聚合和转换日志事件
  └─ 屏蔽浏览器直连 Gateway 的鉴权复杂度
        ↓
OpenClaw Gateway WebSocket
```

## 各层职责

## 1. Browser

浏览器只做两件事。

### HTTP

用于普通请求。

例如。

- 页面初始化
- agent 列表
- session 列表
- overview 数据
- 配置读取
- 配置保存

### 一个统一 WebSocket

用于所有实时能力。

例如。

- chat stream
- tool stream
- logs stream
- session 更新
- presence 更新
- 状态变化

不建议在浏览器侧再拆成多套实时协议。  
也不建议第一版把 chat 用 WS，logs 用 SSE，别的再走轮询。

最简单的方案就是浏览器到 proxy 维护一条统一实时通道。

## 2. Panel Proxy

这层一定要薄。

它不是第二个 OpenClaw。  
只做这几件事。

- 代理浏览器到 Gateway 的请求
- 维护到 Gateway 的 WebSocket 连接
- 统一处理鉴权和连接复用
- 提供浏览器友好的事件格式
- 做日志拉取与转推
- 如果需要，再加少量本地文件和系统动作代理

不要在这层重复实现 OpenClaw 的会话系统、工具系统或业务规则。

## 3. OpenClaw Gateway

仍然是唯一权威控制面。

- chat
- sessions
- skills
- cron
- config
- presence
- logs

这些状态的权威来源仍在 Gateway。  
proxy 只是适配层，不应成为新的状态源。

## 浏览器到 panel proxy 怎么设计

## 普通请求

建议走 HTTP。

例如。

- ​`GET /api/overview`
- ​`GET /api/agents`
- ​`GET /api/sessions`
- ​`GET /api/config`
- ​`POST /api/config/save`

## 实时请求

建议只保留一条浏览器 WebSocket。

浏览器发给 proxy 的消息可以统一成事件式协议。

例如。

- ​`chat.send`
- ​`chat.abort`
- ​`logs.subscribe`
- ​`logs.unsubscribe`
- ​`session.watch`

proxy 返回给浏览器的推送也统一成事件流。

例如。

- ​`chat.started`
- ​`chat.delta`
- ​`chat.done`
- ​`tool.update`
- ​`logs.append`
- ​`logs.reset`
- ​`session.updated`
- ​`presence.update`
- ​`error`

## 为什么推荐浏览器也用 WebSocket

因为这个 panel 明显会做 chat。  
只要 chat 存在，浏览器侧就迟早会有一条实时双向通道。

既然已经有了这条通道，把 logs、session 更新、presence 更新都放进去，复杂度最低。

如果第一版还把日志单独拆成 SSE，就会出现三套心智。

- 普通 HTTP
- 单向 SSE
- 双向 WS

这对后续维护和 vibe coding 都不友好。

## panel proxy 到 OpenClaw 怎么连

这一段直接使用 OpenClaw Gateway WebSocket。

原因很简单。  
OpenClaw 本来就是 Gateway WS 作为控制面。  
proxy 不应该把自己变成一套新的控制协议。

也就是说。

- panel proxy 对外说浏览器友好的协议
- panel proxy 对内仍然说 OpenClaw 原生 Gateway WS 协议

## 实时日志怎么实现

## 第一版最推荐方案

OpenClaw 现有日志能力直接复用 `logs.tail`。

这一点要特别注意。  
OpenClaw 当前的日志展示，不是天然服务器持续推送每一行日志给浏览器的模式。  
官方 Control UI 的日志页，本质上也是通过 `logs.tail` 做 cursor 增量拉取。

所以在 panel proxy 里，最自然的实现方式是。

1. 浏览器发 `logs.subscribe`
2. panel proxy 启动一个轻量轮询任务
3. 每 1 到 2 秒向 OpenClaw Gateway 调一次 `logs.tail`
4. 带上当前 cursor
5. 如果拿到新日志，就通过浏览器 WebSocket 推送 `logs.append`
6. 如果需要重置视图或 cursor 失效，就推 `logs.reset`

这样前端看到的是实时日志流。  
底层实现则仍然复用 OpenClaw 原生能力。

## 为什么这个方案最合适

- 不需要给 OpenClaw 发明新的日志协议
- 不需要 sidecar 先落日志数据库
- 不需要浏览器直接理解 `logs.tail` 的底层细节
- 浏览器只消费统一实时事件
- proxy 只做很薄的事件翻译层

## 推荐的日志数据流

```text
Browser WS
  └─ logs.subscribe { source: "gateway" }
        ↓
Panel Proxy
  └─ logs.tail(cursor)
        ↓
OpenClaw Gateway
  └─ 返回 lines + cursor
        ↓
Panel Proxy
  └─ 推送 logs.append / logs.reset
        ↓
Browser WS
```

## 后续什么时候再升级

只有在这些需求出现时，才值得把日志层再做重。

- 想把 Gateway logs、system logs、Docker logs、plugin logs 聚合
- 想做多源日志订阅
- 想做权限隔离
- 想做真正的 push-based 多源日志流
- 想做统一索引和长期检索

那时候可以给 panel proxy 再扩。

- ​`/logs/stream` SSE 或 WS
- ​`/logs/sources`
- ​`/logs/tail?source=...`

但这属于 phase 2，不应该成为 MVP 的前提。

## 最推荐的落地方式

## 标准模式

适合本机访问，或者能上 HTTPS 的环境。

- Browser 直连 OpenClaw Gateway WS
- 结构更原生
- 组件更少

## 局域网 HTTP 模式

适合你现在这个场景。

- Browser 连接 panel proxy
- panel proxy 再连接 OpenClaw Gateway
- 浏览器侧还是普通网页体验
- 实时能力统一走一条 WebSocket

## 最终判断

如果要做局域网 HTTP 访问的 OpenClaw panel，最佳选择是。

- 保留 OpenClaw Gateway 作为唯一控制面
- 在浏览器前面加一个很薄的 panel proxy
- 普通数据走 HTTP
- 所有实时能力走一条浏览器 WebSocket
- 日志由 proxy 通过 `logs.tail` 增量拉取后转推

这样既保住架构清晰度，也不会被浏览器安全上下文和 Gateway 直连细节拖住。
