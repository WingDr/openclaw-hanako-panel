# OpenClaw Panel 最小可行架构

更新日期 2026-03-22

这份文档专门收束 OpenClaw Panel 的最小可行架构。  
目标是把前面的方案明显缩小，只保留最主要的功能，先做一个可以稳定跑起来、方便继续扩展的 MVP。

## 一句话结论

先不要做完整 panel。  
先做一个最小的 web 面板，只实现三件事。

- 分 agent 的 chat
- 实时 log 监控
- agent 和渠道状态监控

技术上先收缩成：

- 前端 `React + TypeScript + Vite + React Router + Zustand + TailwindCSS`
- 中间层 `panel proxy`
- 后端仍然是 `OpenClaw Gateway`

先不做复杂主题系统。  
先不做 config、cron、skills、文件管理。  
先把最主要的使用闭环做出来。

## 一、MVP 的明确边界

## 只做什么

### 1. 分 agent 的 chat

前端需要支持：

- 选择 agent
- 查看该 agent 下的 session 列表
- 进入某个 session 聊天
- 新建一个属于该 agent 的独立 session
- 看到流式回复

### 2. 实时 log 监控

前端需要支持：

- 查看 Gateway 日志
- 自动滚动
- 暂停滚动
- 简单搜索
- 简单 level 过滤

### 3. agent 和渠道状态监控

前端需要支持：

- Gateway 在线状态
- agent 状态
- channel 状态
- 最近 session 活动概览

## 明确先不做什么

为了把体量真正压下来，这一版先不做：

- config 编辑
- cron 管理
- skills 管理
- models 管理
- 文件管理
- 多用户权限
- session pin / tag / archive
- 命令面板
- split screen
- 复杂右侧上下文侧栏
- 完整 design token 系统

## 二、最小技术栈

## 前端

建议最小前端组合：

- React
- TypeScript
- Vite
- React Router
- Zustand
- TailwindCSS

## 为什么先不要上 TanStack Query

因为这一版页面少，功能边界很窄。  
HTTP 数据获取量也不大。

先自己封装一个轻量 `api.ts` 就够了。  
等页面和接口都稳定之后，如果状态同步复杂度真的上来，再补 TanStack Query。

## 日志组件建议

第一版先不要接完整的大日志组件库。  
先做一个简单日志面板。

只要支持：

- 滚动
- follow
- filter
- search
- buffer 裁剪

如果后面发现性能不够，再切到 `@melloware/react-logviewer`。

## panel proxy

建议保持：

- Node.js
- Fastify
- ​`@fastify/websocket`

第一版甚至可以不落数据库。  
如果只是保存很少量本地状态，先用内存或简单 JSON 文件都够。

## 四、最小数据流

这一版的数据流只保留三条主线。

## 1. chat 数据流

浏览器通过 proxy 发聊天请求。  
proxy 转发给 OpenClaw Gateway。  
流式事件再由 proxy 推回浏览器。

### 浏览器到 proxy

- ​`chat.send`
- ​`chat.abort`
- ​`session.create`
- ​`session.open`

### proxy 到浏览器

- ​`chat.started`
- ​`chat.delta`
- ​`chat.done`
- ​`chat.error`

## 2. logs 数据流

proxy 内部轮询 `logs.tail`。  
浏览器只接收日志追加事件。

### 浏览器到 proxy

- ​`logs.subscribe`
- ​`logs.unsubscribe`

### proxy 到浏览器

- ​`logs.append`
- ​`logs.reset`

## 3. status 数据流

状态页不要求每个指标都强实时。  
可以分成两层。

### 准实时

- agent 状态
- channel 状态
- recent sessions

方式：

- 页面首次通过 HTTP 拉取
- 之后每 5 到 10 秒刷新一次
- 或由 proxy 定时推送 `status.snapshot`

### 实时

- gateway 连接状态

方式：

- 通过 WebSocket 直接推送 `system.connection`

## 五、最小接口设计

## HTTP 接口

第一版只保留这些。

- ​`GET /api/bootstrap`
- ​`GET /api/agents`
- ​`GET /api/agents/:agentId/sessions`
- ​`GET /api/status`
- ​`GET /api/logs/snapshot`

### 每个接口的作用

#### `GET /api/bootstrap`

返回前端启动必须信息。

例如：

- gateway 是否连通
- 默认 agent
- 是否启用 logs 页面
- 基础 feature flags

#### `GET /api/agents`

返回 agent 列表。

#### `GET /api/agents/:agentId/sessions`

返回某个 agent 下的 session 列表。

#### `GET /api/status`

返回状态页总览数据。

#### `GET /api/logs/snapshot`

日志页初始打开时拉一批最近日志。  
后续增量再走 WebSocket。

## WebSocket 命令

第一版只保留这些。

- ​`chat.send`
- ​`chat.abort`
- ​`session.create`
- ​`session.open`
- ​`logs.subscribe`
- ​`logs.unsubscribe`

## WebSocket 事件

第一版只保留这些。

- ​`chat.started`
- ​`chat.delta`
- ​`chat.done`
- ​`chat.error`
- ​`logs.append`
- ​`logs.reset`
- ​`system.connection`
- ​`status.snapshot`

## 六、前端最小 store 结构

这一版 Zustand 一份 store 就够。

推荐拆成四块：

- ​`ui`
- ​`chat`
- ​`logs`
- ​`status`

示意：

```ts
type AppState = {
  selectedAgentId: string | null
  selectedSessionKey: string | null

  agents: AgentSummary[]
  sessionsByAgent: Record<string, SessionSummary[]>

  chatMessages: ChatMessage[]
  chatStreamingText: string

  logs: LogLine[]
  logsFollow: boolean
  logsFilter: string
  logsLevels: string[]

  gatewayConnected: boolean
  statusSnapshot: StatusSnapshot | null
}
```

先不要拆得太细。  
先保证容易理解和容易调试。

## 七、最小 session 设计

这一版独立 session 先只做一种形式：

- ​`agent:<agentId>:panel:<slug>`

不要一开始就做：

- task session
- workspace session
- archive
- pin
- tags

只做到：

- 能创建
- 能打开
- 能继续聊

就足够。

## 八、proxy 侧的最小模块

proxy 内部建议先只做 4 个模块。

## 1. `gatewayClient`

负责：

- 连接 OpenClaw Gateway
- 发送 RPC
- 接收 event

## 2. `browserWsHub`

负责：

- 管理浏览器 WebSocket 连接
- 接收命令
- 分发事件

## 3. `logsService`

负责：

- 轮询 `logs.tail`
- 推送 `logs.append`

## 4. `statusService`

负责：

- 定时拉 agent/channel/session 状态
- 输出 `status.snapshot`

这样最小闭环就成立了。

## 十、建议开发顺序

## 第一步

先把前端静态骨架搭起来。

- 三个页面
- 基础布局
- 假数据

## 第二步

接 `GET /api/agents`​ 和 `GET /api/status`

让 `/chat`​ 和 `/status` 先活起来。

## 第三步

接 chat WebSocket。

做到：

- 发消息
- 收流
- 切 agent
- 切 session

## 第四步

接 logs。

做到：

- 初始快照
- subscribe
- append
- follow

## 第五步

再补基础细节。

- 搜索
- 过滤
- reconnect
- loading 状态
- 错误提示

## 十一、最终推荐

如果现在就要落地，我建议就定成下面这个最小架构。

### 前端只保留三个页面

- ​`/chat`
- ​`/logs`
- ​`/status`

### 前端只保留最小栈

- React + TypeScript + Vite + React Router + Zustand + TailwindCSS

### proxy 只保留四个模块

- ​`gatewayClient`
- ​`browserWsHub`
- ​`logsService`
- ​`statusService`

### 功能只保留三个主能力

- 分 agent chat
- log 监控
- agent/channel 状态监控

## 结论

这版架构已经明显比前面的完整方案小很多。  
而且是真正可以很快做出雏形的一版。

它的价值在于：

- 先把最主要的使用闭环跑通
- 先验证 chat、logs、status 三条主线是否顺
- 给后续扩展 config、cron、skills、完整 manage 面板留下空间

一句话说：  
这不是最终版 panel。  
这是 OpenClaw Panel 真正适合起步的 MVP。
