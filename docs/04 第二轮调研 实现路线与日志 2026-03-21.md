# 2026-03 OpenClaw Panel 第二轮调研

*生成时间：2026-03-21 | 来源数：17 | 置信度：High*

## 摘要

这轮调研的目标有三件事。第一，继续扩大参考项目池，不只看 OpenClaw dashboard，也看更像 openhanako 的 AI workbench、desktop agent 和 agent console。第二，结合 OpenClaw 自身控制架构，决定 panel 最适合采用什么实现架构，尤其考虑后续要走纯 vibe coding，所以要优先选成熟、资料多、 AI 生成代码最稳定的方案。第三，单独研究实时日志展示怎么做，避免一开始就为了日志额外造一层复杂后端。

结论很明确。最合适的方向仍然是 **web-only panel**，并且保持 **chat 与 manage 分路由**。前端建议使用 **React + TypeScript + Vite**，路由层选 **React Router**，状态层用 **Zustand**，服务端数据获取与缓存用 **TanStack Query**，样式采用 **TailwindCSS** 配合少量自定义组件。这个组合不是最“新”，但对 vibe coding 最友好，因为范式稳定、参考项目多、调试成本低。OpenClaw 的核心控制面本来就是 Gateway WebSocket，所以 chat、session、tool stream、基础状态都应该尽量 **直连 Gateway WS**，而不是先包一层自定义业务后端 ([OpenClaw Gateway Protocol](https://docs.openclaw.ai/gateway/protocol), [OpenClaw Control UI](https://docs.openclaw.ai/web/control-ui))。

实时日志方面，不建议第一版额外做 SSE 日志服务。OpenClaw 现有 Control UI 已经通过 `logs.tail`​ 拉取 Gateway 文件日志，官方文档也明确说明 Logs 页就是对 Gateway file logs 的 live tail ([OpenClaw Logging](https://docs.openclaw.ai/gateway/logging), [OpenClaw Control UI](https://docs.openclaw.ai/web/control-ui))。因此 panel 第一版直接复用这条链路就够了。更具体地说，**chat 区的实时性依赖 Gateway chat events**，**manage/logs 页的实时性依赖** **​`logs.tail`​**​ **的 cursor 增量拉取**。如果后面要把 sidecar、自定义系统日志、容器日志一起并进来，再考虑用 sidecar 暴露 SSE 或 WebSocket 聚合日志流。

## 研究问题

本轮围绕三个子问题展开。

1. 除了 OpenClaw 专用 dashboard，还有哪些更像 openhanako 的工作台或 agent app 值得参考。
2. Panel 最适合直接对接 OpenClaw 的哪一层，前端技术栈应该怎么定，才能兼顾成熟度和 vibe coding 体验。
3. 实时日志展示应该复用 OpenClaw 现有能力，还是自建新的日志传输层。

## 一、参考项目扩展结果

## 1.1 与 openhanako 同气质的 workbench / desktop agent 类

### openhanako

openhanako 仍然是这类项目里最接近目标气质的参考。它强调 personal AI agent、memory、personality、autonomy，并且前端是 Electron + React + Vite，适合借鉴整体视觉、工作台气质、agent 存在感和 onboarding 方式，但不适合直接搬它的内部 manager 体系 ([openhanako](https://github.com/liliMozi/openhanako))。

### Zuckerman

Zuckerman 是一个 ultra-minimal personal AI agent，强调 agent 自修改、热更新、 plain text 配置、 CLI + Electron/React 双界面。它很像一种更偏“极简 agent OS”的思路，适合借鉴两点：一是工作台和可视化界面的轻壳思路，二是把 agent 定义、工具、身份和对话拆成清晰目录的方式 ([Zuckerman](https://github.com/zuckermanai/zuckerman))。

### ai-browser

ai-browser 是 Electron + Next.js 的 desktop automation agent，重点在智能浏览器、多模态、计划执行、设置面板和 workflow 模式。它对 panel 的启发不是 Electron，而是“把聊天、深度任务模式、历史、设置分成独立页面”的产品分层方式，以及完整设置系统通过 UI 配置而不是手改文件的思路 ([ai-browser](https://github.com/DeepFundAI/ai-browser))。

### CodePilot

CodePilot 是 Claude Code 的桌面 GUI，核心关键词是 chat、code、manage projects visually。它和你要做的东西最接近的点在于：一个 assistant workspace、可暂停恢复的 session、split-screen、多 provider、MCP、skills 和 bridge。它说明“chat 工作台”和“项目/系统管理”可以共存，但一定要靠页面结构把它们分开，不然会越做越重 ([CodePilot](https://github.com/op7418/CodePilot))。

### Goose

Goose 更像 on-machine AI agent 产品。虽然公开页面里 UI 细节不多，但它同时提供 desktop app 和 CLI，也强调 multi-model、MCP、可扩展性。它说明 agent 产品里“本地执行 + 可视化外壳”是成立的，但它更偏执行器，不像 openhanako 那样强调人格化工作台 ([Goose](https://github.com/block/goose))。

## 1.2 平台/控制台类参考

### VoltAgent / VoltOps Console

VoltAgent 提供框架，VoltOps Console 提供控制台。它的价值不在于 UI 风格，而在于它把 chat、workflow、trace、memory、eval、health、deployment 分层得很清楚。对于你的 panel 来说，它提示了一件事：如果以后要扩成更完整的 observability/workflow 平台，应该把 traces、logs、memory、sessions 设计成独立视图，而不是都往一个 dashboard 里堆 ([VoltAgent](https://github.com/VoltAgent/voltagent))。

## 1.3 OpenClaw 专用面板参考

### ClawPanel

ClawPanel 是 OpenClaw 的管理外壳。它的后端做了 JWT 认证、REST API 封装、配置和进程级管理，更像系统管理台而不是原生会话工作台 ([ClawPanel](https://github.com/zhaoxinyi02/ClawPanel), [ClawPanel API](https://github.com/zhaoxinyi02/ClawPanel/blob/main/docs/API.md))。

### openclaw-agent-monitor

它通过轮询 `openclaw status --json`​ 和会话 `.jsonl`​ 做全量可视化监控，适合借鉴 overview 页、总览卡片和轻部署，但不适合作为主控制架构，因为它更偏旁路观测，而不是原生控制面 ([openclaw-agent-monitor](https://github.com/vvlang/openclaw-agent-monitor))。

### openclaw-manager

它走 Tauri + React + Rust，更偏桌面服务管理器。适合借鉴设置页、诊断页和结构化配置方式，但不适合直接作为 web-only panel 的主范式 ([openclaw-manager](https://github.com/miaoxworld/openclaw-manager))。

### openclaw-dashboard

它是最完整的产品型 dashboard 参考，覆盖 sessions、costs、live feed、memory、files、logs、cron、security 等。适合借鉴路由组织方式和 manage 分区，但也最容易把产品引向“重运维后台” ([openclaw-dashboard](https://github.com/tugcantopaloglu/openclaw-dashboard))。

## 1.4 从这些项目里提炼出的四种产品形态

|形态|代表项目|强项|风险|
| ------------------| -------------------------------| -------------------------------------------| ------------------------------------|
|人格化 workbench|openhanako, Zuckerman|气质强，chat/workspace 感好|容易把内部 runtime 一起搬过来|
|桌面 agent app|ai-browser, CodePilot, Goose|对话、任务、设置分层完整|容易被 Electron/Next.js 复杂度拖重|
|平台 console|VoltOps Console|traces、logs、workflow、memory 的分区成熟|容易越做越像 B 端控制台|
|OpenClaw 管理台|ClawPanel, openclaw-dashboard|对 OpenClaw 结构贴近|容易把运维接管做得过重|

## 二、对 OpenClaw 控制架构的判断

## 2.1 Gateway WebSocket 才是 OpenClaw 的真正控制面

OpenClaw 官方文档写得很清楚，Gateway WS 是 **single control plane + node transport**。CLI、Web UI、移动端节点、headless node 都连这个 WebSocket，并在握手里声明 role 和 scopes ([OpenClaw Gateway Protocol](https://docs.openclaw.ai/gateway/protocol))。

这件事直接决定了 panel 的技术方向。

如果 panel 的主要工作是：

- 聊天
- 管理 session
- 看 tool stream
- 看状态
- 调 config / skills / cron 这些 Gateway 暴露的方法

那么最自然的实现不是自建一套业务后端，而是前端直接做一个 Gateway WS client。

## 2.2 官方 Control UI 已经证明“浏览器直连 Gateway”可行

官方 Control UI 是一个 **Vite + Lit** 的 SPA，并且明确说明它直接连 Gateway WebSocket，同端口工作 ([OpenClaw Control UI](https://docs.openclaw.ai/web/control-ui))。

它今天已经覆盖了：

- ​`chat.history`
- ​`chat.send`
- ​`chat.abort`
- ​`chat.inject`
- ​`sessions.list`
- ​`sessions.patch`
- ​`cron.*`
- ​`skills.*`
- ​`config.get`
- ​`config.set`
- ​`config.apply`
- ​`system-presence`
- ​`logs.tail`
- ​`update.run`

也就是说，绝大多数 panel 必需能力，本来就在 Gateway 上。

## 2.3 Session 的所有权在 Gateway，不在 UI

OpenClaw session 文档明确说明，session state 由 Gateway 持有，UI 客户端要向 Gateway 查询 session list 和 token counts，而不是直接去读本地文件 ([OpenClaw Session Management](https://github.com/openclaw/openclaw/blob/main/docs/concepts/session.md))。

这对 panel 很关键。因为这意味着：

- 独立 session 是可以做的
- 但 session transcript/token/context 的权威来源应始终是 Gateway
- panel 只应该补充 UI 元数据，比如 title、tag、pin、archived 状态

## 2.4 ACP 说明 OpenClaw 天然支持“非频道来源的会话”

​`openclaw acp`​ 的作用是把 ACP session 映射到 Gateway session key，并且默认就有隔离的 `acp:<uuid>`​，也支持显式指定 `agent:main:main`​、`agent:qa:bug-123`​ 这类 key ([OpenClaw ACP](https://github.com/openclaw/openclaw/blob/main/docs/cli/acp.md))。

这意味着 panel 自己创建独立 session 是完全顺理成章的。它不依赖 Telegram、QQ、Discord。它只依赖稳定的 session key 设计。

## 三、Panel 应该采用什么实现架构

## 3.1 结论

最推荐的是 **A-first with optional B**。

### A

前端直接连接 Gateway WebSocket。

### B

只有在需要本地文件、系统动作、额外日志、宿主机诊断时，再补一个很薄的 sidecar。

我不建议一开始就做 C。

### C

完整自建业务后端，再由后端去包装 OpenClaw。

这样会重复构建 OpenClaw 已经提供的控制层，既变重，也更不适合 vibe coding。

## 3.2 为什么这套架构最适合纯 vibe coding

纯 vibe coding 最怕两件事。

第一，跨层抽象太多。前端一层、业务后端一层、OpenClaw Gateway 再一层，出问题时很难定位。  
第二，框架组合太花。SSR、RPC、服务端 actions、桌面壳、本地数据库、进程控制一起上，模型写出来的代码会越来越不稳定。

所以 panel 要故意选“ boring but durable ”的技术路线。

## 3.3 推荐技术栈

### 前端

- React
- TypeScript
- Vite
- React Router
- Zustand
- TanStack Query
- TailwindCSS
- 可选 `@radix-ui/react-*`​ 或 `shadcn/ui` 风格组件

### 传输层

- 主链路：Gateway WebSocket
- 次链路：可选 sidecar REST / SSE / WebSocket

### 可选 sidecar

- Node.js
- 一个很薄的 HTTP 服务
- 只处理本地文件、白名单系统动作、额外日志聚合

## 3.4 为什么不是 Next.js

Next.js 当然能做，但这次不推荐。

原因不是它不成熟，而是它解决的问题和这个 panel 的核心问题不一致。

- 这个 panel 不需要 SEO
- 首屏 SSR 价值不高
- 核心通信是长连 WS，而不是服务端渲染页面
- 如果再叠加本地 sidecar，会形成两套服务端心智

ai-browser 和 CodePilot 之所以用 Next.js，和它们的 desktop hybrid、复杂设置页面、产品演进路径有关 ([ai-browser](https://github.com/DeepFundAI/ai-browser), [CodePilot](https://github.com/op7418/CodePilot))。但你的 panel 目标更贴近 OpenClaw 官方 Control UI 的浏览器控制面，所以 **Vite 更顺手，也更贴合原生模式** ([OpenClaw Control UI](https://docs.openclaw.ai/web/control-ui), [openhanako](https://github.com/liliMozi/openhanako))。

## 3.5 为什么不是继续跟官方 Lit

官方 Control UI 用的是 Lit，这本身没问题。但如果目标是纯 vibe coding，我仍然更推荐 React。

原因很简单。

- 参考项目绝大多数是 React
- 现成组件生态更丰富
- AI 生成 React + Tailwind + Zustand 代码的稳定性明显更高
- 你后面想融 openhanako 风格，也天然更顺

所以这不是否定 Lit，而是对“实现效率与后续维护成本”的判断。

## 3.6 路由结构建议

panel 维持两大主域。

- ​`/chat`
- ​`/manage`

建议的页面树如下。

```text
/chat
/chat/:agentId/:sessionKey

/manage/overview
/manage/agents
/manage/sessions
/manage/channels
/manage/models
/manage/skills
/manage/cron
/manage/logs
/manage/config
```

这里最关键的是把 **工作线程** 和 **系统管理** 拆开。

- ​`/chat` 负责 session、stream、tool call timeline、上下文工作
- ​`/manage` 负责 overview、列表、筛选、配置、诊断

这个结构同时吸收了 openhanako 的工作台感、openclaw-dashboard 的 manage 分路由、ClawPanel 的管理视角 ([openhanako](https://github.com/liliMozi/openhanako), [openclaw-dashboard](https://github.com/tugcantopaloglu/openclaw-dashboard), [ClawPanel](https://github.com/zhaoxinyi02/ClawPanel))。

## 3.7 独立 session 设计建议

panel 应该直接支持 agent 级独立 session。

推荐 key 形式。

- ​`agent:<agentId>:panel:<slug>`
- ​`agent:<agentId>:task:<slug>`
- ​`agent:<agentId>:workspace:<slug>`

比如。

- ​`agent:main:panel:design-review`
- ​`agent:main:task:panel-mvp`
- ​`agent:research:workspace:paper-notes`

这和 OpenClaw 已有的 session key 风格一致，也和 ACP 文档中的 `agent:qa:bug-123`​ 一脉相承 ([OpenClaw ACP](https://github.com/openclaw/openclaw/blob/main/docs/cli/acp.md), [OpenClaw Session Management](https://github.com/openclaw/openclaw/blob/main/docs/concepts/session.md))。

## 四、实时日志展示怎么实现

## 4.1 第一结论

**第一版不要额外造新的日志传输层。**

OpenClaw 已经有 `logs.tail`​。官方文档写明了 Control UI 的 Logs tab 就是通过 Gateway 来 tail file logs ([OpenClaw Logging](https://docs.openclaw.ai/gateway/logging), [OpenClaw Control UI](https://docs.openclaw.ai/web/control-ui))。

所以 panel 第一版可以直接采用同样的模式。

## 4.2 关键区别：chat 的实时性和 logs 的实时性不是一回事

这件事很容易混。

### chat / tool stream

应该走 Gateway 的 chat events。  
这是真正的流式对话和工具执行流。

### logs

应该走 `logs.tail`。  
这是 Gateway 文件日志视图，不是聊天事件流。

也就是说，panel 里至少应该有两种“实时”。

1. 对话实时
2. 日志实时

这两者不要强行用同一套渲染模型。

## 4.3 当前官方实现方式值得直接借鉴

从本地安装包里的 Control UI 代码可以看出，Logs 页是通过调用 `logs.tail`​ 并基于 cursor 轮询更新的。代码里默认每 2 秒刷新一次，使用 `limit`​、`maxBytes`​、`cursor`​ 做增量拉取，并在前端做 level filter、文本过滤、auto-follow 与导出。Overview 页的 log tail 则是更轻的一版，拉更小的窗口，只展示最近片段（本机 OpenClaw `dist/control-ui`​ 内置实现，配合 [OpenClaw Logging](https://docs.openclaw.ai/gateway/logging) 与 [CLI Gateway](https://docs.openclaw.ai/cli/gateway) 对 `logs.tail` 的说明）。

这说明官方已经替你验证了一个很务实的方案。

- 不需要单独日志订阅协议
- 不需要 sidecar 先转一次
- 不需要先落地数据库
- 直接 cursor 增量拉取即可

## 4.4 第一版日志页推荐做法

### 数据获取

- 在 `/manage/logs`​ 使用现有 Gateway WS client 发 `logs.tail`
- 使用 cursor 做增量拉取
- 轮询间隔 1 到 2 秒
- 只保留最近固定窗口，比如 500 到 2000 行

### UI 能力

- level filter
- 文本搜索
- auto-follow 开关
- pause / resume
- export current view
- ANSI 颜色解析
- 大量日志时的虚拟滚动

### 为什么这样最好

因为它与 OpenClaw 官方能力一致，成本最低，而且足够满足 panel 第一版的“实时日志展示”需求。

## 4.5 日志视图组件建议

最适合直接借鉴的是 `@melloware/react-logviewer`。

它支持：

- 懒加载和大文件展示
- 虚拟滚动
- ANSI 高亮
- EventSource / WebSocket / URL / streamed response
- follow 模式
- 搜索 ([react-logviewer](https://github.com/melloware/react-logviewer))

它本身是从 `react-lazylog`​ 演进而来，而 `react-lazylog`​ 这条路线已经验证了“浏览器展示超大日志文本 + ANSI + follow 模式”的可行性 ([react-lazylog](https://github.com/mozilla-frontend-infra/react-lazylog), [react-logviewer](https://github.com/melloware/react-logviewer))。

所以有两个可行路径。

### 路径 A

直接接入 `@melloware/react-logviewer` 作为日志渲染层。

优点是快。  
纯 vibe coding 也容易让模型生成可工作的接线代码。

### 路径 B

自己做一个简化版日志列表。

- 虚拟列表
- ANSI parser
- 自动跟随
- 顶部过滤器

优点是主题统一性更好。  
缺点是首版开发量更高。

我的建议是：**先 A 后 B**。

## 4.6 什么时候才需要 sidecar 日志流

只有在下面这些需求出现时，才值得补 sidecar 日志流。

- 想把 Gateway logs、system logs、Docker/container logs、plugin logs 聚合到一个视图
- 想做真正的多源订阅式日志
- 想做更低延迟的 push 模式，而不是轮询
- 想做日志权限隔离和多种源的统一索引

这时可以让 sidecar 提供：

- ​`/logs/stream` SSE
- ​`/logs/sources` 列出可订阅源
- ​`/logs/tail?source=gateway` 获取快照

但这应该是 **phase 2**，不是 MVP。

## 五、推荐实现方案

## 5.1 最终推荐

### 产品层

- web-only
- chat 与 manage 分路由
- 统一主题
- 不做 Electron

### 前端层

- React + TypeScript + Vite
- React Router
- Zustand
- TanStack Query
- TailwindCSS

### 传输层

- Gateway WebSocket 作为主链路
- ​`logs.tail` 作为日志页主方案
- 可选薄 sidecar 作为后续扩展

### 核心功能顺序

#### 第一阶段

- ​`/chat`
- session 列表
- 新建独立 session
- chat send / stream
- tool stream

#### 第二阶段

- ​`/manage/overview`
- ​`/manage/agents`
- ​`/manage/sessions`
- ​`/manage/logs`

#### 第三阶段

- ​`/manage/channels`
- ​`/manage/models`
- ​`/manage/skills`
- ​`/manage/cron`
- ​`/manage/config`

#### 第四阶段

- sidecar 文件代理
- sidecar 系统动作
- 聚合日志
- 更强运维页

## 5.2 不推荐的方案

### 不推荐一开始做完整自建后端

因为会重复 OpenClaw 的控制层。

### 不推荐第一版就 Next.js

因为 SSR 价值低，复杂度高。

### 不推荐第一版就 Electron

因为真正难的不是桌面壳，而是信息架构和数据流。

### 不推荐为实时日志单独发明协议

因为 `logs.tail` 已经足够起步。

## 六、核心结论

这轮调研之后，方向已经可以定了。

### 结论 1

panel 该做成 **OpenClaw 原生工作台**，而不是另一个重运维后台。

### 结论 2

从 openhanako、CodePilot、ai-browser、VoltOps Console 和几种 OpenClaw dashboard 参考看，最佳产品结构就是：

- chat 独立成工作区
- manage 独立成管理区
- 两者统一主题

### 结论 3

从 OpenClaw 官方架构看，panel 最应该直连的是 Gateway WebSocket，而不是先包装一层自建控制后端 ([OpenClaw Gateway Protocol](https://docs.openclaw.ai/gateway/protocol), [OpenClaw Control UI](https://docs.openclaw.ai/web/control-ui))。

### 结论 4

从 vibe coding 的成熟度和稳定性考虑，最推荐的技术组合是：

- React + TypeScript + Vite
- React Router
- Zustand
- TanStack Query
- TailwindCSS

这是当前最不容易把自己绕进去的路线。

### 结论 5

实时日志第一版直接用 `logs.tail`，配合 React log viewer 或简化版虚拟日志列表即可。真正需要 sidecar 日志服务的时候，再进入第二阶段。

## 研究方法

- 第一轮检索：先用 `searxng_search`​ 做广义发现，因结果质量不足，再用 `web_search` 进行定向 GitHub 检索。
- 第二轮精读：对关键项目 README 和 OpenClaw 本地文档逐项核对。
- 本轮子问题：

  - 哪些项目更像 openhanako 这种 workbench 而不是纯 dashboard
  - OpenClaw 的控制面应该直连哪一层
  - web-only 的 panel 应该选择什么框架组合
  - 实时日志应该复用什么现有能力
- 信息缺口 / 不确定性：

  - 某些第三方项目没有公开完整架构文档，所以对内部实现深度的判断部分基于 README 和公开页面，属于中等置信度。
  - OpenClaw Control UI 当前日志轮询细节来自本机安装包实现与官方文档交叉判断，属于高置信度，但具体轮询参数未来版本可能变化。
