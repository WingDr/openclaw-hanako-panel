# OpenClaw Control Panel 调研结论

更新日期 2026-03-21

这份文档重新整理了已验证资料。  
旧的 `OpenClaw-Control-Panel-Development-Guide.md` 只能当草稿参考，这一版才作为结论稿。

## 一句话结论

能做。  
而且最好不要把 ClawPanel 的做法当成唯一范式。  
如果目标是做一个更像 openhanako 风格的 OpenClaw control panel，最合理的路线是前端界面学 openhanako，控制链路尽量直接对接 OpenClaw Gateway WebSocket，只有宿主机管理这类脏活再交给一个很薄的本地后端。

独立 session 也能做。  
它本质上不是频道问题，而是 session key 设计问题。  
OpenClaw 现有模型已经支持一个 agent 对应多个 session，只要面板主动生成稳定的 agent 级 session key，就能得到不依赖 Telegram QQ Discord 之类通道的独立上下文。

## OpenClaw 的调用原理

## 1. 真正的控制平面是 Gateway WebSocket

OpenClaw 的核心不是某个 REST API 服务，而是 Gateway。  
Gateway 使用 WebSocket 作为统一控制平面。  
CLI、浏览器 Control UI、移动端节点、其他客户端，都是连到这个 Gateway。

协议上分三类消息。

- request
- response
- event

握手流程大致是这样。

1. 客户端先收到 `connect.challenge`
2. 客户端带着 token 和 device 签名发 `connect`
3. Gateway 返回 `hello-ok`
4. 后续所有控制动作都走同一个 WS 连接

这意味着，如果你自己做 control panel，最贴近 OpenClaw 原生架构的方式，不是再包一层业务后端，而是直接做一个 Gateway WS 客户端。

## 2. 官方 Control UI 本身就是直连 Gateway

OpenClaw 官方文档已经写得很明确。  
内置的 Control UI 是一个 Vite + Lit 的单页应用，直接连 Gateway WebSocket，同端口工作。

它现在已经能调这些能力。

- ​`chat.history`
- ​`chat.send`
- ​`chat.abort`
- ​`chat.inject`
- ​`sessions.list`
- ​`sessions.patch`
- ​`cron.*`
- ​`skills.*`
- ​`config.get`
- ​`config.set`
- ​`config.apply`
- ​`system-presence`
- ​`logs.tail`
- ​`update.run`

所以从原理上说，做 OpenClaw panel 有两条路线。

### 路线 A

直接仿官方 Control UI。  
前端自己连 Gateway WS。

优点是链路最短，最原生。  
缺点是浏览器侧要处理 device auth、pairing、token、断线重连、WS 事件状态机。

### 路线 B

仿 ClawPanel。  
自己再包一层 REST 后端。  
前端只打自己的 REST 和 WS。

优点是前端简单，宿主机管理能力也容易塞进去。  
缺点是会把 OpenClaw 原本的实时协议再包装一遍，维护成本更高。

如果你的目标更偏 OpenClaw 专用 control panel，不是大而全运维面板，我更建议路线 A 或 A+B 混合。

## 3. ACP 不是主控制接口，但很适合独立会话接入

​`openclaw acp` 的作用是把 ACP 客户端映射到 Gateway session。  
它通过 stdio 对外说 ACP，再转发到 Gateway WebSocket。

关键点不是 ACP 协议本身，而是它暴露了一个很重要的事实。

一个 ACP session 可以绑定到一个明确的 Gateway session key。  
官方示例里就有。

- ​`agent:main:main`
- ​`agent:design:main`
- ​`agent:qa:bug-123`
- 默认隔离的 `acp:<uuid>`

这说明 OpenClaw 的 session 并不要求必须来自某个聊天平台。  
只要你能给它一个合法且稳定的 session key，就能把 agent 上下文挂起来。

## ClawPanel 到底是怎么调用 OpenClaw 的

ClawPanel 不是 OpenClaw 原生前端。  
它更像一个 OpenClaw 管理外壳。

从它公开的 API 文档看，它做了几层事情。

### 1. 自己先做了一层 JWT 认证 REST API

例如。

- ​`/api/auth/login`
- ​`/api/system/status`
- ​`/api/agents`
- ​`/api/channels`
- ​`/api/skills`
- ​`/api/sessions`
- ​`/api/system/config`

这代表它的前端并不直接调用 OpenClaw Gateway WS。  
而是先调用 ClawPanel 自己的 Go 后端。

### 2. 它包装的是 OpenClaw 配置和运行时状态

从 API 设计能看出来，ClawPanel 很多能力并不是聊天协议层，而是运维层。

比如。

- 读写 `openclaw.json`
- 管理 agent 配置
- 管理 bindings
- 管理 skills 和 plugins
- 管理 cron jobs
- 查看 sessions
- 重启网关
- 检查环境
- 跟 NapCat 和微信等外部组件交互

所以它本质上是。

OpenClaw Gateway + 宿主机进程/配置管理 + 额外渠道集成

这也解释了为什么它需要 Go 后端。  
因为它不只是一个聊天 UI，而是一个控制台和安装器。

### 3. 它对 OpenClaw 的“调用”大概率分三类

第一类是配置文件级。  
直接读写 `openclaw.json` 和工作区核心文件。

第二类是进程级。  
启动、停止、重启 OpenClaw 或相关依赖进程。

第三类才是会话级。  
读取 session store、transcript、agent 信息，然后封装成自己的 REST 数据源给前端。

所以如果你只是想做一个更漂亮的 OpenClaw panel，不必完整复刻 ClawPanel 这整套。  
你可以只拿它的管理维度参考，不必照抄它的后端分层。

## 能不能融合 openhanako 的前端样式

能。  
而且技术上是顺手的。

## 1. 技术栈并不冲突

openhanako 公开信息里是。

- Electron 38
- React 19
- Vite 7
- React 前端

ClawPanel 前端是。

- React 18
- TypeScript
- Vite
- TailwindCSS

OpenClaw 官方 Control UI 虽然是 Vite + Lit，不是 React，但这不构成阻碍。  
因为你本来就是要做一个新的 panel，不是给官方 UI 打补丁。

也就是说，你完全可以这样做。

- UI 容器和视觉体系学 openhanako
- 数据层直接对接 OpenClaw Gateway
- 需要宿主机操作时接一个很薄的本地服务

## 2. 真正能复用的是视觉语言，不是源码直接拼装

我不建议把 openhanako 前端源码直接硬塞到 OpenClaw 上。  
理由很简单。

它的数据模型不是按 OpenClaw Gateway 协议设计的。  
它内部还有自己的 Agent Manager、Session Manager、Desk、Memory 系统。  
强行直接嫁接，后面会被数据适配层拖死。

更合理的做法是复用这些部分。

- 布局结构
- 导航层级
- 卡片风格
- 对话区质感
- onboarding 流程感
- agent 工作台的整体氛围

不要直接复用这些部分。

- 内部状态管理假设
- 数据实体定义
- session 生命周期逻辑
- memory 和 desk 的底层实现

一句话说，就是学皮相和交互，不搬内脏。

## 独立 session 能不能做

能。  
而且这是这件事里最值得做的功能。

## 1. 从 OpenClaw 的 session 模型看，这件事天然成立

OpenClaw 的 session 文档明确说明，session 的本体是 `sessionKey -> sessionId` 的映射。  
状态由 Gateway 持有。  
不同来源只是映射到不同 key。

文档里已经有这些来源。

- direct chat
- group chat
- cron
- webhook
- node runs
- ACP

这里最重要的是，OpenClaw 已经承认“不是聊天平台来源的 session”这件事。  
例如。

- ​`cron:<job.id>`
- ​`hook:<uuid>`
- ​`node-<nodeId>`
- ​`acp:<uuid>`

所以 panel 自己造一个前缀是合理的。

## 2. 最实用的设计是 agent 级 panel session

我建议不要发明太激进的新 schema。  
直接沿着 agent session key 的风格扩展。

推荐格式。

- ​`agent:<agentId>:panel:<sessionSlug>`
- ​`agent:<agentId>:workspace:<workspaceId>`
- ​`agent:<agentId>:task:<taskId>`

例如。

- ​`agent:main:panel:architecture-review`
- ​`agent:research:panel:paper-notes`
- ​`agent:main:task:openclaw-control-panel`

这样有几个好处。

第一，肉眼可读。  
第二，天然知道归属哪个 agent。  
第三，和官方示例 `agent:qa:bug-123` 的风格一致。  
第四，不依赖 channel sender peer accountId。

## 3. 这个独立 session 的本质是什么

本质上是给 panel 一个“主动创建上下文”的入口。

不是等 Telegram 或 QQ 来消息才生成上下文。  
而是用户在面板里点一下。

- 选 agent
- 填 session 名称
- 新建上下文
- 面板以后持续向这个 session key 发 `chat.send`

这样就能得到真正的面板内工作线程。

它适合这些场景。

- 长期项目线程
- 调研线程
- 代码审阅线程
- 仅 agent 自己可见的工作上下文
- 不希望污染主聊天上下文的实验线程

## 4. 需要注意的边界

### 多客户端共用同一个 session key 时，事件路由只是 best effort

ACP 文档明确提到。  
如果多个 ACP 客户端共享同一个 Gateway session key，事件和 cancel 路由不是强隔离。

这个提示也适用于 panel 设计。  
所以默认应该一线程一 key。  
不要让两个面板标签页同时抢同一个活跃会话，除非用户明确知道自己在干什么。

### 会话隔离是 key 级，不是 UI tab 级

不要把“开了三个标签页”误以为“有三个 session”。  
必须显式创建三个不同 session key 才算独立上下文。

### 会话持久化本来就由 Gateway 负责

这一点反而是好事。  
面板不用自己发明一套会话存储。  
只需要保存 session 的展示元数据，比如。

- title
- icon
- pinned
- lastOpenedAt
- userTags

真正的 transcript 和 token 统计，本来就在 Gateway。

## 推荐架构

## 方案一 纯前端直连 Gateway

适合做最像 OpenClaw 原生 Control UI 的版本。

结构。

- React 前端
- openhanako 风格 UI
- 直接连接 OpenClaw Gateway WS

负责内容。

- chat
- sessions
- tools 流输出
- config 读取
- presence
- skills 和 cron 的读操作

优点是轻。  
缺点是宿主机管理能力弱。

## 方案二 前端直连 Gateway，加一个很薄的本地管理服务

这是我最推荐的。

结构。

- React 前端负责 OpenClaw 原生交互
- 本地 sidecar 负责宿主机事务

sidecar 只处理这些事情。

- 打开本地文件
- 导入导出
- 调系统命令
- 收集额外日志
- 本地缓存 UI 偏好
- 如果需要，再代理一部分认证

这样做的好处是分层干净。

OpenClaw 相关会话和工具流，还是走原生 Gateway。  
宿主机脏活，单独隔离。

## 方案三 做成 Electron 桌面应用

如果你想让它更像 openhanako，本地桌面感更强，可以直接上 Electron。

这时可以做成。

- Electron 壳
- React 前端
- 内置 WS 客户端
- 可选内嵌 sidecar

优点是体验完整。  
缺点是打包和更新复杂度会立刻上来。

如果一开始只是验证产品方向，我反而建议先做 Web 版，再决定要不要套 Electron。

## 为什么我不建议直接复刻 ClawPanel

因为 ClawPanel 的价值密度很大一部分来自“运维接管”。

它除了 OpenClaw，还管。

- NapCat
- 微信桥
- 安装更新
- 服务重启
- 配置备份
- 宿主机巡检

如果你的目标是一个更漂亮、更顺手、更偏 agent 工作台的 panel，这些东西会迅速把项目做重。

真正该先做的是三件事。

- 原生 chat 与 tool stream
- agent 级独立 session
- openhanako 风格工作台

这三件做好，产品辨识度就出来了。

## MVP 建议

第一版只做下面这些。

### 必做

- agent 列表
- session 列表
- 新建独立 session
- 对指定 session 聊天
- 会话流式输出
- tool call 时间线
- 读取基础状态

### 第二阶段

- session pin 和标签
- 对 session 的 rename 和归档
- 每个 session 绑定 workspace 或 task
- agent 身份卡和技能概览

### 第三阶段

- 宿主机管理 sidecar
- config 编辑
- cron 面板
- 插件与技能管理

## 最终判断

如果你要的是一个真正属于 OpenClaw 的 control panel，我的判断是。

可行。  
方向也对。

但最佳实现不是 ClawPanel 翻版。  
而是。

- 交互体验学 openhanako
- 控制链路学 OpenClaw 官方 Control UI
- 管理维度参考 ClawPanel
- 独立 session 作为核心卖点之一

这样做出来的东西会更轻，更原生，也更有差异化。

## 本文对应的验证点

这次结论依赖这些已验证事实。

1. OpenClaw 官方 Control UI 直连 Gateway WS
2. Gateway WS 是统一控制平面
3. session 状态由 Gateway 持有
4. ACP 可以把会话映射到明确的 Gateway session key
5. OpenClaw session 不只服务于聊天平台来源
6. ClawPanel 公开 API 明显是一层自建管理后端
7. openhanako 的前端技术栈与新 panel 的实现方向兼容
